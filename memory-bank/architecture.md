## 1. 设计目标

Word Formatter 采用**前后端分离**的架构思想。整个软件划分为以下核心部分：

- **UI**（WinUI 3）
- **API 通信层**
- **Python 后端**（业务逻辑 + 排版引擎）
- **数据存储**

各层职责严格独立，互不耦合。任何一层均不得直接访问其他层的内部实现，所有交互均通过定义良好的接口完成。

---

## 2. 整体架构

系统自顶向下分为五个逻辑层：

```
┌────────────────────────────────────────────┐
│          WinUI 3 (Presentation)            │
│          UI + MVVM                         │
└──────────────────┬─────────────────────────┘
                   │ HTTP REST API (JSON)
┌──────────────────▼─────────────────────────┐
│          FastAPI (API Layer)               │
│          请求分发、参数校验、响应封装         │
└──────────────────┬─────────────────────────┘
                   │
┌──────────────────▼─────────────────────────┐
│          Service Layer                     │
│          业务逻辑、任务调度、配置管理         │
└──────────────────┬─────────────────────────┘
                   │
┌──────────────────▼─────────────────────────┐
│          Formatter Engine                  │
│          python-docx 排版核心               │
└──────────────────┬─────────────────────────┘
                   │
┌──────────────────▼─────────────────────────┐
│          Data & Storage                    │
│          文件 / 模板 / 历史 / 配置            │
└────────────────────────────────────────────┘
```

**核心原则**：上层只能向下依赖，下层不得反向调用上层。

## 3. 分层职责

### 3.1 UI 层（WinUI 3）

- 负责页面渲染、用户交互、动画、文件拖拽与选择、配置编辑展示、状态栏显示、任务进度反馈。
- **禁止**：直接操作 Word 文档、读写原始排版数据、调用任何后端排版逻辑。

### 3.2 ViewModel 层（MVVM）

- 承担数据绑定、命令（Command）封装、页面状态与控件状态管理（如当前模板、文件数量、任务状态、按钮可用性）。
- **禁止**：直接操作 python-docx、文件系统或 Word 排版引擎。

### 3.3 API 层（FastAPI）

- 接收 WinUI 发起的 HTTP 请求，进行参数校验，调用对应的 Service，并以 JSON 格式返回处理结果。
- **不含业务逻辑**，仅做透传和基本验证。

### 3.4 Service 层

- 实现所有核心业务：文件管理、模板管理、历史记录、排版任务调度、配置管理。
- 是系统的**唯一业务入口**，任何上层请求都必须经过 Service 层协调。

### 3.5 Formatter Engine（排版引擎）

- 基于 `python-docx` 等库实现具体的 Word 排版操作：页面设置、页眉页脚、正文样式、标题样式、图片与表格处理、文档保存。
- **不得**：弹窗、直接更新 UI、写日志到界面。

### 3.6 Data Layer（数据层）

- 负责持久化存储：配置文件、模板 JSON、历史记录、最近打开列表、缓存等。
- **不得**：包含任何 UI 逻辑或排版逻辑。

## 4. 模块划分

系统功能模块划分如下：

```
Word Formatter
├── UI (WinUI 3 Views & Controls)
├── File Manager          # 文件管理（添加、扫描、过滤）
├── Profile Manager       # 配置方案管理
├── Template Manager      # 模板的增删改查、导入导出
├── Preview Engine        # 排版预览（参数摘要或真实预览）
├── Formatter Engine      # 实际排版执行
├── History Manager       # 历史任务记录
├── Config Manager        # 全局设置管理
└── Logger                # 统一日志记录
```

各模块职责清晰，通过接口协作，无交叉依赖。

## 6. 通信与数据流

### 6.1 通用请求流程

```
用户操作 (UI)
   → ViewModel 更新状态
   → API Service 发起 HTTP 请求
   → FastAPI 接收请求
   → Service 处理业务
   → 返回 JSON 结果
   → ViewModel 解析并更新 UI
```

### 6.2 排版任务流程

```
用户点击“开始排版”
   → API 发送排版请求（携带文件列表与配置）
   → Formatter Service 调度任务
   → 逐个文件调用 Formatter Engine
   → python-docx 执行排版并保存
   → 返回每个文件的处理结果（成功/失败）
   → UI 实时更新进度与结果统计
```

### 6.3 数据流向约束

- UI **不允许直接**调用 Python 后端，必须通过 HTTP API。
- ViewModel **不能**引用 `python-docx` 或任何排版相关模块。
- Formatter Engine **不能**引用 WinUI 相关命名空间或进行 UI 通知。

## 7. UI 布局与页面组成

软件采用**固定三栏布局**，所有页面遵循统一结构：

```
┌───────────────────────────────────────────────────────┐
│ 标题栏（Title Bar）                                    │
├──────────┬──────────────────────────┬──────────────────┤
│ 导航栏   │ 配置区域                  │ 文件与排版控制    │
│ (Nav)    │ (Configuration)          │ (File & Format)  │
├──────────┴──────────────────────────┴──────────────────┤
│ 状态栏（Status Bar）                                   │
└───────────────────────────────────────────────────────┘
```

- **左栏（导航）**：列出所有功能模块（页面设置、正文样式、标题样式、页眉页脚、图片设置、表格设置、高级设置）。
- **中栏（配置）**：根据当前选中的导航项显示对应的设置界面。
- **右栏（文件与排版）**：包含文件管理、输出目录、排版任务控制与进度展示。
- **底栏（状态）**：显示当前状态、文件数量、模板名称等实时信息。

## 8. 配置与数据存储

系统配置与用户数据统一存放在后端 `config/` 目录下：

```
config/
├── settings.json        # 全局软件设置（如语言、输出目录偏好）
├── templates/           # 所有排版模板（JSON 文件）
├── history/             # 历史任务记录（SQLite / JSON）
├── logs/                # 运行日志
└── cache/               # 临时或缓存文件
```

模板文件采用 JSON 格式，包含版本号以便兼容性检查。历史记录支持按日期查询与复用。

## 9. 日志规范

日志记录统一由 `Logger` 模块（`backend/utils/logger.py`）处理，接口为
`get_logger(name, category)`。采用**按用途分文件 + 按日期自动归档**：既便于按场景
排查问题，又保留历史归档，两者兼得。

**按用途分文件**（活动文件位于 `logs/` 根目录）：

| 日志文件 | 覆盖内容 |
| --- | --- |
| `app.log` | 前端 / UI：软件启动与关闭、主题切换 |
| `backend.log` | API 请求与响应、Service 业务逻辑、文件导入与扫描 |
| `format.log` | 排版任务：开始、进度、完成、失败、逐文件处理 |
| `error.log` | 所有异常（ERROR 及以上），跨全部分类统一汇总 |

`error.log` 为横切汇总：无论日志属于哪个分类，只要级别 ≥ ERROR 都会额外写入一份，
便于集中排查异常，同时仍保留在其原分类文件中。

**按日期自动归档 + 自动保留**（`TimedRotatingFileHandler`，每日午夜滚动）：活动文件
午夜滚动后在 `logs/` 根目录留下按日期命名的 `{分类}-YYYY-MM-DD.log`。保留策略分两级，
**用户无需手动清理**：

| 位置 | 保留 | 到期处理 |
| --- | --- | --- |
| `logs/`（根目录每日文件） | 最近 30 天 | 超过 30 天移入 `logs/archive/` |
| `logs/archive/` | 最近 90 天 | 超过 90 天自动删除 |

清扫在进程启动首次取 logger 时执行一次，并在每次午夜滚动后再执行一次。示例：

```
logs/
├── app.log                       # 当天活动文件
├── backend.log
├── format.log
├── error.log
├── app-2026-07-11.log            # 近期每日文件（≤30 天，留在根目录）
├── backend-2026-07-11.log
└── archive/
    ├── app-2026-05-30.log        # 归档（31~90 天），超 90 天自动删除
    └── backend-2026-05-30.log
```

日志级别 INFO，格式 `[时间] [级别] [模块名] 消息内容`。日志仅在后端产生，前端通过
API 获取必要的错误信息，不直接写入日志文件。禁止 `print()`，统一使用 `logging`。

## 10. 设计原则与约束

| 原则       | 说明                                          |
| -------- | ------------------------------------------- |
| **单一职责** | 每个模块只负责一件事，不越界。                             |
| **高内聚**  | 模块内部功能关联紧密，对外暴露最小接口。                        |
| **低耦合**  | 模块间仅通过 API 或明确接口通信，禁止直接调用内部实现。              |
| **可扩展**  | 新增格式（如 PDF、Excel、Markdown）不应影响现有代码，只需扩展相应层。 |
| **可维护**  | 任何功能均可独立测试、替换或升级。                           |
| **可替换**  | UI 层可替换为 Avalonia、Web 等技术，后端无需修改。           |

**强制约束**：

- UI 不处理业务逻辑。
- ViewModel 不操作 Word 文档。
- API 层仅负责协议转换、参数验证和响应封装，不承载任何业务逻辑或排版实现。
- Service 不直接操控 UI。
- Formatter 不依赖 WinUI。
- 所有业务逻辑进入 Service，所有 Word 操作进入 Formatter，所有配置进入 Config，所有模板进入 Template Manager，所有日志进入 Logger。

## 11. 后续扩展预留

系统架构预留了以下扩展方向，新增功能可通过增加相应的 Service 或模块实现，无需修改核心排版引擎：

- 多语言国际化
- 深色模式 / 主题切换

- 自动更新机制

所有扩展必须遵循分层原则，保持现有架构的清晰性与稳定性。
